== Feature Flags

[%notitle]
=== what are feature flags?

Feature Flags [.step.highlight-green]#enable, disable or change the behavior# of certain features or code paths in a product or service, [.step.highlight-green]#without modifying the source code.#

[.notes]
--
* Env vars
* config flags
--

=== Why?

=== Coordinate And Target

* Synchronized rollouts
* Experiments
* Usergroup specific features

=== Reduce Risk

* Deployment != Release
* Risk-averse releases
* Progressive rollouts

=== Categories of Feature Flags

* differ in longevity and dynamism
* different needs

=== Description

OpenFeature is an [.step.highlight-red]#open specification# that provides a [.step.highlight-red]#vendor-agnostic, community-driven API for feature flagging# that works with your favorite feature flag management tool.
footnote:[https://openfeature.dev/docs/reference/intro]

=== basic usage - java

[source, java, highlight="1,2|4|6-7" step=0]
----
OpenFeatureAPI api = OpenFeatureAPI.getInstance();
api.setProviderAndWait(new InMemoryProvider(myFlags));

Client client = api.getClient();

boolean flagValue = 
    client.getBooleanValue("v2_enabled", false);
----
footnote:[https://openfeature.dev/docs/reference/technologies/server/java]

[transition=fade-in,fade-out]
=== basic usage - node.js

[source, javascript, highlight]
----
import { OpenFeature } from '@openfeature/server-sdk';
await OpenFeature.setProviderAndWait(new YourProviderOfChoice());

const client = OpenFeature.getClient();

const v2Enabled = 
    await client.getBooleanValue('v2_enabled', false);
----
footnote:[https://openfeature.dev/docs/reference/technologies/server/javascript]


[transition=fade-in,slide-out]
=== basic usage - golang

[source, golang]
----

openfeature.SetProvider(openfeature.NoopProvider{})

client := openfeature.NewClient()

v2Enabled, _ := client.BooleanValue(
    context.Background(), 
    "v2_enabled", 
    true, 
    openfeature.EvaluationContext{},
)
----
footnote:[https://openfeature.dev/docs/reference/technologies/server/go]

=== FlagD - Targeting Example

[source, json, highlight="3|5-8|10-19" step=0]
----
{
  "flags": {
    "v2_enabled": {
      "state": "ENABLED",
      "variants": {
        "on": true,
        "off": false
      },
      "defaultVariant": "off",
      "targeting": { 
        "if": [
          {
            "ends_with": [
              { "var": "email" }, "@domain.com"
            ]
          },
          "on",
          "off"
        ]
      }
    }
  }
}
----
[.social]
a simple flag config with targeting


=== Dynamic Context

[source, java, highlight="1-5|6-7|9-10" step=0 ]
----
Map<String, Value> requestAttrs = new HashMap<>();
requestAttrs.put("email", 
    new Value(session.getAttribute("email")));
requestAttrs.put("product", 
    new Value("productId"));
EvaluationContext reqCtx = 
    new ImmutableContext(requestAttrs);

boolean flagValue = 
    client.getBooleanValue("v2_enabled", false, reqCtx);
----

== Obsolete Feature Flags

[%notitle]
=== what?

Feature Flags that only evaluate to the same value anymore.

=== Why?

* Dead code
* Technical debt
* Increased complexity

=== Decommissioning

* Remove outdated features
* Remove obsolete behaviour
* Remove complexity

=== ...but when is it safe?

=== Flag Evaluation Life-Cycle
--
image::evallifecycle.svg[link="https://openfeature.dev/specification/sections/hooks#overview",caption="",title="",background="none"]
--
// [mermaid, theme="neutral", background="000000"]
// ----
// flowchart LR
//     B("'Before' stage")
//     FE["Flag Evaluation"]
//     E("'Error' stage")
//     A("'After' stage")
//     F("'Finally' stage")
//     B ---> FE
//     B -..-> |"Error"| E
//     FE ---> A
//     FE -..-> |"Error"| E
//     A ---> F
//     E -..-> F
// ----
// footnote:[https://openfeature.dev/specification/sections/hooks#overview]

[transition=slide-in,fade-out]

Boolean value = client.getBooleanValue(

    );
